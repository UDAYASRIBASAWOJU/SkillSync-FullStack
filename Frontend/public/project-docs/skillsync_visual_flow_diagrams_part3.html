⚡ TanStack Query + Axios — Data Fetching Strategy

apiHooks.ts — useQuery + useMutation patterns
PATTERN: useQuery wraps Axios calls and provides: → Automatic caching (client-side, in memory) → Background refetching (stale-while-revalidate) → Loading/error states → Deduplication (same queryKey = one request even if multiple components mount) EXAMPLE — Admin stats: const { data, isLoading, error } = useAdminStats(); → useQuery({ queryKey: ['admin', 'stats'], queryFn: () => api.get('/api/admin/stats') }) First render: isLoading=true → spinner shown Data arrives: data=AdminStatsResponse → dashboard renders Next mount of same component: cache hit → instant render, background refetch EXAMPLE — Mutations (write operations): const loginMutation = useMutation({ mutationFn: async (data) => api.post('/api/auth/login', data), onSuccess: (data) => { dispatch(setCredentials(...)); navigate(roleBasedPath); }, onError: (error) => { showToast({ message: error.response?.data?.message, type: 'error' }); } }); onSubmit = (data) => loginMutation.mutate(data); CACHE INVALIDATION: After a mutation (e.g., approving a mentor), TanStack Query cache must be invalidated: queryClient.invalidateQueries({ queryKey: ['admin', 'mentors', 'pending'] }); → Forces background refetch on next access → No stale data shown to admin HOW IT LAYERS WITH AXIOS: TanStack Query → manages when to call Axios → manages how to call (interceptors, auth headers, error handling) Redux → manages persistent state (auth, role) across component unmounts

⚙️ Payment Saga State Machine

PaymentStatus Enum — All States & Transitions (payment-service)
PAYMENT STATUSES: CREATED → Razorpay order created, awaiting user payment VERIFIED → Razorpay signature verified (payment received) SUCCESS_PENDING → Saga started: business action events queued in outbox SUCCESS → All saga steps completed successfully FAILED → Signature invalid / Razorpay error COMPENSATED → Saga failed and rollback was triggered STATE FLOW: createOrder(): CREATED ──────────────────────────────────────────→ (awaiting payment) verifyPayment() — signature check passes: CREATED → VERIFIED → [saga starts] → SUCCESS_PENDING → SUCCESS verifyPayment() — signature check fails: CREATED → FAILED → publishPaymentFailedEvent() → outbox → "payment.failed.v1" → session-service: rollbackSessionPayment() saga fails mid-execution: SUCCESS_PENDING → COMPENSATED (after max retries exhausted by SagaRecoveryScheduler) SAGA STATE TABLE (saga_state table): { paymentId, orderId, state=SUCCESS_PENDING, retryCount=0, lastError, createdAt } → Each saga step persists new state to DB → SagaRecoveryScheduler reads this table to find stuck sagas DUPLICATE PAYMENT PREVENTION: preventDuplicatePayment(): → paymentRepository.findByReferenceIdAndReferenceTypeAndStatusIn( referenceId, referenceType, [CREATED, VERIFIED, SUCCESS_PENDING]) → IF EXISTS → throw PaymentException("DUPLICATE_PAYMENT", 409 CONFLICT) → "An active payment already exists for this reference" → Forces client to cancel existing payment before creating a new one

🔁 SagaRecoveryScheduler — Stuck Saga Healing

SagaRecoveryScheduler.java — @Scheduled every 5 minutes
PROBLEM: What if the service crashes AFTER writing SUCCESS_PENDING to the saga table but BEFORE the OutboxPublisher publishes the event to RabbitMQ? → Payment is verified, money taken, but session is never confirmed → Saga is STUCK in SUCCESS_PENDING forever SOLUTION: SagaRecoveryScheduler @Scheduled(fixedDelayString = "${saga.recovery.interval-ms:300000}") ← every 5 min @Transactional public void recoverStaleSagas(): LocalDateTime threshold = now().minusMinutes(10); ← saga.recovery.timeout-minutes=10 List staleSagas = sagaStateRepository.findStaleSagas( PaymentStatus.SUCCESS_PENDING, threshold) → Finds all sagas stuck in SUCCESS_PENDING for > 10 minutes FOR EACH stale saga: recoverSaga(saga): → Payment payment = paymentRepository.findById(saga.paymentId) IF saga.retryCount >= maxRetries(3): ← saga.recovery.max-retries=3 compensateSaga(saga, payment): → payment.status = COMPENSATED → saga.state = COMPENSATED → outboxEventService.saveEvent("payment.compensated.v1", compensatedEvent) → sagaCompensationCounter.increment() ← Micrometer metric ELSE (retry): → Build PaymentCompletedEvent with new UUID eventId → outboxEventService.saveEvent("payment.business.action.v1", retryEvent) → saga.retryCount++ → sagaStateRepository.save(saga) → sagaRecoveryCounter.increment() ← Micrometer metric METRICS TRACKED (Micrometer → Prometheus): saga.recovery.attempts → count of retries attempted saga.recovery.compensations → count of sagas auto-compensated (money kept, session cancelled) @PostConstruct void initMetrics(): → Registers Counter beans with MeterRegistry on startup → Available at /actuator/prometheus for Grafana scraping OUTCOME: → Max 3 retries over 15+ minutes before auto-compensation → Compensation = payment COMPENSATED + session CANCELLED → User notified via notification-service (payment.compensated.v1 event) → Micrometer metrics allow ops team to detect systematic saga failures

💡
ProcessedEvent Table — Event Consumer Idempotency The processed_events table in user-service stores eventIds of all handled RabbitMQ messages. Before processing, processedEventRepository.existsByEventId(eventId) is called. If the event was already processed (RabbitMQ redelivered due to crash before ACK), it's silently skipped. This is the standard pattern for at-least-once message delivery in production systems.

🗃️ RedisConfig Deep-Dive — Serialization Choices

RedisConfig.java (skillsync-cache-common) — Every Decision Explained
RedisTemplate configuration: KEY SERIALIZER: StringRedisSerializer → Keys stored as plain strings: "v1:user:profile:101" → Human-readable in Redis CLI: redis-cli KEYS "v1:*" → Why String not default JDK serialization? Default adds binary prefix, making keys unreadable and SCAN pattern matching unreliable VALUE SERIALIZER: GenericJackson2JsonRedisSerializer (custom ObjectMapper) Why Jackson over default JDK serialization? → JDK serialization: binary, class-version-sensitive, Java-only → Jackson JSON: human-readable, version-tolerant, cross-language CUSTOM ObjectMapper settings: → mapper.registerModule(new JavaTimeModule()) → Handles Java 8+ time types: LocalDateTime, Instant, Duration → Without this, LocalDateTime fields throw UnsupportedTypeException → mapper.disable(SerializationFeature.WRITE_DATES_AS_TIMESTAMPS) → Stores dates as ISO-8601 strings ("2024-01-15T10:30:00") not timestamps (1705316400000) → Human-readable, avoids timezone confusion → mapper.activateDefaultTyping(LaissezFaireSubTypeValidator.instance, ObjectMapper.DefaultTyping.NON_FINAL) → CRITICAL for deserialization! → Stores the actual Java class name alongside the JSON: ["com.skillsync.user.dto.ProfileResponse", { "userId": 101, ... }] → Without this: Jackson can't know what class to deserialize into → ClassCastException when reading from Redis → LaissezFaireSubTypeValidator: allows any class (less strict, more compatible) → Security note: in production, use a whitelist validator instead → mapper.setVisibility(PropertyAccessor.ALL, JsonAutoDetect.Visibility.ANY) → Allows Jackson to access private fields directly → Needed for Lombok @Data classes where fields are private HASH SERIALIZERS: → hashKeySerializer = StringRedisSerializer → hashValueSerializer = GenericJackson2JsonRedisSerializer → Used for HASH type Redis operations (opsForHash) → Not currently used by CacheService (uses STRING type), but configured for extensibility RESULT: Cache entries look like this in Redis: Key: "v1:user:profile:101" Value: ["com.skillsync.user.dto.ProfileResponse", {"userId":101,"firstName":"John","lastName":"Doe","bio":"...","skills":[...]}]

⚡ Cache Stampede Protection — Deep Explanation

CacheService.getOrLoad() — Double-Check Lock Pattern
PROBLEM: Cache Stampede (aka "Thundering Herd") Scenario: Cache key "v1:user:profile:101" expires at exactly 12:00:00 → 500 concurrent requests all miss the cache simultaneously → All 500 call profileRepository.findByUserId(101) at the same time → DB gets hammered with 500 identical queries → could crash or slow to a crawl SOLUTION: Double-Check Locking with ReentrantLock (CacheService.java) private final ConcurrentHashMap keyLocks = new ConcurrentHashMap<>(); public T getOrLoad(String key, Class type, Duration ttl, Supplier dbFallback): STEP 1: Try cache (no lock — fast path) ────────────────────────────────────────── T cached = get(key, type); if (cached != null) return cached; ← 499 of 500 requests return here if (isNullSentinel(key)) return null; ← "we know it doesn't exist" STEP 2: Acquire per-key lock ────────────────────────────────────────── ReentrantLock lock = keyLocks.computeIfAbsent(key, k -> new ReentrantLock()); lock.lock(); ← All 500 threads compete. ONE wins. 499 block here. STEP 3: Double-check inside lock ────────────────────────────────────────── try { cached = get(key, type); ← The winner checks AGAIN if (cached != null) return cached; ← 499 losing threads return here if (isNullSentinel(key)) return null; ← after winner populates cache STEP 4: DB query (only ONE thread reaches here) T value = dbFallback.get(); ← profileRepository.findByUserId(101) if (value != null) put(key, value, ttl); ← Cache populated by winner else putNull(key); ← Null-sentinel: prevents cache penetration return value; } finally { lock.unlock(); keyLocks.remove(key, lock); ← Cleanup to prevent memory leak } NULL SENTINEL PATTERN: If a user profile genuinely doesn't exist → DB returns null Without null-sentinel: every request misses cache → every request hits DB With null-sentinel: cache stores "__NULL__" with 60s TTL → Next 60s of requests see null-sentinel → return null immediately, no DB hit → Prevents "cache penetration" (requests for non-existent data always hit DB) METRICS: cacheHitCounter → "cache.operations" tag result=hit, service=user-service cacheMissCounter → "cache.operations" tag result=miss, service=user-service cacheEvictCounter → "cache.operations" tag result=evict cacheErrorCounter → "cache.operations" tag result=error (Redis connection failure) → All available at /actuator/prometheus → Grafana dashboard

📅 JPA Auditing — Auto-Timestamps

JpaAuditingConfig.java + @EnableJpaAuditing — How created_at/updated_at Work
EVERY MICROSERVICE has: JpaAuditingConfig.java: @Configuration @EnableJpaAuditing public class JpaAuditingConfig {} ← Just this annotation, no extra code needed ENTITY FIELDS (e.g., AuthUser.java): @CreatedDate @Column(updatable = false) private LocalDateTime createdAt; → Set automatically on first INSERT → never updated again (updatable=false) @LastModifiedDate private LocalDateTime lastModifiedAt; → Updated automatically on every UPDATE @EntityListeners(AuditingEntityListener.class) ← required on the entity class → Spring Data JPA hooks: beforeInsert(), beforeUpdate() → Calls AuditingEntityListener which reads the CurrentAuditor WHAT HAPPENS UNDER THE HOOD: 1. For @CreatedDate: sets field to LocalDateTime.now() (on new entities) 5. For @LastModifiedDate: sets field to LocalDateTime.now() (on every save) 6. Entity is persisted with timestamps populated — zero developer effort WHY NOT JUST SET TIMESTAMPS MANUALLY? → Developers forget → inconsistent data → Clock differences between services → different formats → @EnableJpaAuditing centralizes and standardizes all timestamp management → createdAt is immutable (updatable=false) → can never be accidentally overwritten

❓ Advanced Interview Q&A — Part 3

Redis & Caching

QuestionAnswer (from actual code)
Why does RedisConfig use activateDefaultTyping()?Without it, redisTemplate.opsForValue().get(key) returns a LinkedHashMap (raw JSON), not the typed ProfileResponse. Jackson needs the @class metadata embedded in JSON to reconstruct the correct Java class during deserialization. The alternative is using RedisTemplate<String, ProfileResponse> — one template per type — which is unscalable.
What is the Null Sentinel pattern?When dbFallback.get() returns null (entity not found), putNull(key) stores "__NULL__" with 60s TTL. Next 60 seconds of requests for that key return null immediately without hitting DB. Without this, every request for a non-existent entity would be a cache miss → DB query → still null. Called "cache penetration" prevention.
Why use ConcurrentHashMap for key locks instead of a single global lock?A single lock would serialize ALL cache misses across ALL keys — catastrophic for throughput. ConcurrentHashMap gives per-key locks — threads for key A are not blocked by threads for key B. The stampede protection is scoped to each cache key independently.

Saga & Payment

QuestionAnswer (from actual code)
What happens if the OutboxPublisher crashes after writing to the outbox table but before publishing to RabbitMQ?The outbox event stays with published=false. On restart, OutboxPublisher polls SELECT * FROM outbox_events WHERE published=false and republishes. This guarantees at-least-once delivery. The consumer's processedEventRepository.existsByEventId(eventId) check handles the duplicate delivery case.
Why does SagaRecoveryScheduler use @PostConstruct for metrics?MeterRegistry beans are available after the ApplicationContext is fully initialized. @PostConstruct runs after all dependency injection is complete — safe to call Counter.builder(...).register(meterRegistry). Doing this in the constructor risks NullPointerException if the MeterRegistry isn't injected yet.
What is the difference between payment.failed.v1 and payment.compensated.v1 routing keys?payment.failed.v1: Signature verification failed at payment time — immediate failure, no saga started. payment.compensated.v1: Saga started but failed after max retries — compensation triggered by SagaRecoveryScheduler. Both cancel the session, but compensated means money may have been taken (Razorpay verification passed) while failed means the payment was rejected by Razorpay itself.

The One-Paragraph Answer for Any Interview

"SkillSync is a full-stack distributed SaaS platform built on Spring Cloud microservices with a React/Redux frontend. The API Gateway uses Netty with four filter layers — CSRF origin validation, sliding-window rate limiting, security headers, and JWT authentication — routing to six microservices via Eureka. The Auth-Service implements the full Spring Security filter chain with HS256 JWT generation, BCrypt-12 password hashing, 6-digit OTP email verification, Google OAuth implicit flow with account linking, and stateful refresh token rotation with FIFO eviction up to five tokens per user. All business services follow CQRS — command services handle transactional writes with Redis cache eviction and RabbitMQ event publishing via the Outbox Pattern, while query services use a cache-aside strategy with stampede protection via per-key ReentrantLock and null-sentinel penetration prevention. Payment uses a Saga Orchestrator with a SagaRecoveryScheduler that retries stuck sagas on a five-minute schedule and auto-compensates after three failed retries, all tracked via Micrometer counters exported to Prometheus. Real-time notifications are pushed via STOMP WebSocket through a SimpMessagingTemplate, while email notifications use a retry queue with dead-letter fallback. Every consumer uses a ProcessedEvent idempotency table to safely handle RabbitMQ's at-least-once delivery. Cross-service calls use Feign clients targeting internal endpoints blocked at the Gateway, with batch requests used to eliminate N+1 fan-out. The Redis cache uses GenericJackson2JsonRedisSerializer with DefaultTyping for polymorphic deserialization, versioned key prefixes for zero-downtime schema migration, and SCAN-based pattern eviction. Infrastructure uses Spring Cloud Config Server with fail-fast startup, Eureka IP-based registration, multi-stage Docker builds, Zipkin distributed tracing, and structured logging with traceId/spanId correlation."
