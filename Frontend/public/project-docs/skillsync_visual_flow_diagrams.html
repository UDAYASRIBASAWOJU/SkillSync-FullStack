🛡️ Gateway Filter Chain — Detailed Breakdown

OrderFilter ClassTypeLogicFail → Result
-3 CsrfOriginFilter GlobalFilter POST/PUT/DELETE/PATCH: checks Origin header against allowedOrigins (skillsync.udayasri.dev, localhost). OPTIONS skipped. 403 Forbidden immediately
-2 RateLimitingFilter GlobalFilter Sliding window counters. OTP=5/min, Login=10/min, Payment=10/min, Public=60/min, Authenticated=100/min. Key = user_id or IP. ConcurrentHashMap buckets. 429 Too Many Requests + headers
-1 SecurityHeadersFilter GlobalFilter Adds to RESPONSE: X-Frame-Options: DENY, X-Content-Type-Options: nosniff, X-XSS-Protection: 1; mode=block, HSTS, Referrer-Policy. No failure case — always adds headers
route JwtAuthenticationFilter GatewayFilter (per route) Checks Authorization header then accessToken cookie. Validates HMAC-SHA256 signature. Strips spoofed X-User-* headers. Injects: X-User-Id, X-User-Email, X-User-Role. 401 Unauthorized

🎫 JWT Internals — Structure & Lifecycle

JWT Token Structure (actual from JwtTokenProvider.java)
TOKEN = base64(HEADER) . base64(PAYLOAD) . SIGNATURE HEADER (decoded): { "alg": "HS256", ← HMAC-SHA256 symmetric algorithm "typ": "JWT" } PAYLOAD / CLAIMS (decoded, from generateAccessToken()): { "sub": "101", ← userId (String.valueOf(userId)) "email": "user@skillsync.com", ← claims Map.of("email", email, "role", role) "role": "ROLE_LEARNER", "iat": 1700000000, ← issuedAt (new Date()) "exp": 1700086400 ← expiration (now + 24h) } SIGNATURE: HMACSHA256( base64UrlEncode(header) + "." + base64UrlEncode(payload), BASE64.decode(JWT_SECRET) ← From application.properties: jwt.secret ) ACCESS TOKEN: exp = now + max(accessExpiration, 24h) REFRESH TOKEN: exp = now + max(refreshExpiration, 7 days) (only has sub + iat + exp — no email/role claims) SIGNING KEY (JwtTokenProvider.getSigningKey()): byte[] keyBytes = Decoders.BASE64.decode(jwtSecret); return Keys.hmacShaKeyFor(keyBytes); ← javax.crypto.SecretKey VALIDATION (JwtTokenProvider.isTokenValid()): try { Claims claims = Jwts.parser() .verifyWith(getSigningKey()) .build() .parseSignedClaims(token) ← validates signature integrity .getPayload(); return claims.getExpiration().after(new Date()); ← checks expiry } catch (Exception e) { return false; ← invalid signature, expired, malformed → false } KEY EXTRACTION METHODS: extractUserId() → Long.parseLong(claims.getSubject()) extractEmail() → claims.get("email", String.class) (role is not extracted from JWT — loaded fresh from DB via UserDetails)

❓ Interview Q&A — From the Actual Code

Gateway Questions

QuestionAnswer (from actual code)
Why does the Gateway strip X-User-Id before injecting it?To prevent header spoofing. A malicious client could set X-User-Id: 999 to access another user's data. JwtAuthenticationFilter.java: h.remove("X-User-Id") then .header("X-User-Id", claims.getSubject()). Only the verified JWT claim is trusted.
Why use RateLimitingFilter with ConcurrentHashMap instead of Redis?Comment in RateLimitingFilter.java: "in-memory sliding window counters (production: swap for Redis-based)". Current approach works for single-instance but won't work across multiple Gateway instances. Redis would be needed for horizontal scaling.
What happens to /api/auth/internal/** from the public internet?routes[0] in application.properties: Path=/api/auth/internal/** → uri=no://op → SetStatus=403. Gateway returns 403 before any service is reached. Internal services call each other directly bypassing the Gateway.

Auth-Service Questions

QuestionAnswer (from actual code)
Why are there TWO DB calls in AuthService.login()?First call: authenticationManager.authenticate() → internally calls UserDetailsServiceImpl.loadUserByUsername() → DB query for password hash validation. Second call: authUserRepository.findByEmail() to get the full AuthUser entity to check isVerified and generate tokens. This is a known pattern — UserDetails doesn't include all fields.
Why max 5 refresh tokens per user?AuthService.java: MAX_REFRESH_TOKENS = 5. FIFO eviction: findByUserOrderByCreatedAtAsc(user) → delete oldest if count >= 5. Supports multiple devices (phone + laptop + tablet) without unbounded token accumulation.
Why does the JWT filter call loadUserByUsername() on every request?JwtAuthenticationFilter.java line: UserDetails userDetails = userDetailsService.loadUserByUsername(email). This ensures the account is still active and not disabled. The isActive field is checked here. Cost: one DB query per protected request. Could be optimized with Redis UserDetails caching.
How does the SecurityConfig know not to apply auth to /api/auth/login?SecurityConfig.java: .requestMatchers("/api/auth/login", ...).permitAll(). FilterSecurityInterceptor sees this URL matches a permitted pattern and allows without authentication check.

Frontend Questions

QuestionAnswer (from actual code)
Why is the access token stored in BOTH Redux AND localStorage?authSlice.ts: Redux for fast in-memory access by Axios interceptor. localStorage for persistence across page refreshes (Redux resets on refresh). localStorage is read back via getStoredAuth() in initialState.
What is AuthLoader and why is it needed?AuthLoader.tsx: On page refresh, Redux is empty (memory cleared) but HttpOnly cookies still exist. AuthLoader calls GET /api/auth/me (which extracts user from JWT cookie), then dispatches setCredentials to restore Redux state. Without it, every page refresh would force logout.
How does ProtectedRoute work?ProtectedRoute.tsx: useSelector(state => state.auth.isAuthenticated). If false → <Navigate to="/login" replace />. If true → <Outlet /> (renders the protected child route). Works with React Router's nested routing.
Why does Google OAuth flow call Google's userinfo endpoint instead of using ID token?LoginPage.tsx comment: "Since we're using the Implicit Flow with useGoogleLogin, we can either get user info from an ID token (if available) or by calling Google's userInfo endpoint using the access_token." They use the userinfo endpoint approach for simplicity.
