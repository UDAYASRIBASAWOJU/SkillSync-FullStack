⚙️ Session State Machine

Defined in SessionStatus.java using a static Map<SessionStatus, Set<SessionStatus>> of allowed transitions. The Session.isTransitionAllowed(target) method is checked before every status change.

REQUESTED ACCEPTED REJECTED COMPLETED CANCELLED
Allowed State Transitions (SessionStatus.java — ALLOWED_TRANSITIONS Map)
REQUESTED ──→ ACCEPTED (mentor accepts) ──→ REJECTED (mentor rejects) ──→ CANCELLED (learner or mentor cancels / payment fails) ACCEPTED ──→ COMPLETED (mentor marks complete) ──→ CANCELLED (either party cancels) REJECTED ──→ (terminal — no further transitions) COMPLETED ──→ (terminal — no further transitions) CANCELLED ──→ (terminal — no further transitions) VALIDATION CODE (SessionCommandService): private void validateTransition(Session session, SessionStatus target) { if (!session.isTransitionAllowed(target)) { throw new RuntimeException( "Cannot transition from " + session.getStatus() + " to " + target); } } public boolean isTransitionAllowed(SessionStatus target) { return ALLOWED_TRANSITIONS.getOrDefault(this, Set.of()).contains(target); } EACH TRANSITION: accept() → validateTransition(ACCEPTED) → save → invalidateCaches → publishEvent("session.accepted") reject() → validateTransition(REJECTED) → save → invalidateCaches → publishEvent("session.rejected") cancel() → checks mentorId OR learnerId ownership → validateTransition(CANCELLED) → publishEvent complete() → mentorId ownership only → validateTransition(COMPLETED) → defaultRatingApplied=true → publishEvent("session.completed") → publishMentorMetricsUpdate(mentorId) → RabbitMQ "review.summary.updated"

🔄 CQRS Pattern — Used Across All Services

Every microservice (user, session, notification, skill, payment) follows CQRS — Command Query Responsibility Segregation. Commands and queries are in separate classes.

CQRS Architecture — user-service Example
CONTROLLERS: UserController → UserCommandService + UserQueryService MentorController → MentorCommandService + MentorQueryService GroupController → GroupCommandService + GroupQueryService AdminController → uses both QUERY SERVICES (READ operations): UserQueryService.java → getProfile(userId), getProfileById(id), searchUsers() MentorQueryService.java → getMentorById(), searchMentors(), getPendingApplications() GroupQueryService.java → getGroup(), listGroups(), searchGroups() All query services use: → CacheService.getOrLoad() — Redis-first, DB fallback → ProfileRepository.findByUserId() (read-only transactions) COMMAND SERVICES (WRITE operations): UserCommandService.java → createOrUpdateProfile(), addSkill(), removeSkill() MentorCommandService.java → apply(), approve(), reject(), addAvailability() GroupCommandService.java → createGroup(), addMember(), postDiscussion() All command services: → @Transactional (full ACID guarantees) → Cache eviction after write: cacheService.evict() or evictByPattern() → RabbitMQ event publishing for side effects → Feign calls for cross-service data (authServiceClient, skillServiceClient) WHY CQRS? → Read traffic >> Write traffic in this app (mentors read often, write occasionally) → Query services can be optimized independently (Redis caching, read replicas) → Command services can have heavier validation without slowing down reads → Clear separation of concerns: each class has ONE responsibility → Easier to test: mock the repository in unit tests per service type SAME PATTERN IN session-service: SessionCommandService → createSession, acceptSession, rejectSession, completeSession SessionQueryService → getSessionById, getSessionsByLearner, getSessionsByMentor ReviewCommandService → submitReview, deleteReview ReviewQueryService → getMentorReviews, getMentorRatingSummary, getMyReviews

📬 Outbox Pattern — At-Least-Once Delivery

Why Outbox? — Dual Write Problem Solution
THE PROBLEM (Dual Write): paymentRepository.save(payment); ← DB commit rabbitTemplate.convertAndSend(...); ← RabbitMQ publish If RabbitMQ is down when DB succeeds → event LOST forever. If DB fails after RabbitMQ publish → event published but no payment record. This leads to inconsistent state. THE SOLUTION — Outbox Pattern: @Transactional public void verifyPayment(...) { payment.setStatus(COMPLETED); paymentRepository.save(payment); ← STEP 1: Update payment OutboxEvent event = OutboxEvent.builder() ← STEP 2: Write event to SAME DB .eventType("PAYMENT_COMPLETED") .payload(json(paymentCompletedEvent)) .published(false) .build(); outboxEventRepository.save(event); ← Same transaction = atomic! } // If EITHER save fails → BOTH roll back → consistent state OutboxPublisher (scheduled/polled): @Scheduled (or triggered on DB commit) → SELECT * FROM payment.outbox_events WHERE published=false → For each: rabbitTemplate.convertAndSend(exchange, routingKey, payload) → event.published = true → outboxEventRepository.save(event) → AT-LEAST-ONCE: if publisher crashes mid-way, unprocessed events stay in outbox with published=false → retried on next poll SagaRecoveryScheduler.java: → Scheduled recovery job in payment-service → Finds SagaState records stuck in intermediate states → Triggers retry or compensation for hung sagas → Handles edge cases: Razorpay webhook timeout, consumer restart, etc. DLQ (Dead Letter Queue): → email.retry.queue has x-dead-letter-exchange configured → After N retries (EmailRetryConsumer), message goes to email.dlq → DlqReplayController.java (in payment-service): POST /api/payments/dlq/replay → replays failed events

🚨 Exception Handling — Two Completely Different Paths

Exception Flow — Filter Layer vs Controller Layer
PATH A: EXCEPTION IN FILTER LAYER (before DispatcherServlet) ═══════════════════════════════════════════════════════════════ Example: JWT expired inside JwtAuthenticationFilter.doFilterInternal() JwtAuthenticationFilter: → jwtTokenProvider.isTokenValid(token) returns false (ExpiredJwtException) → Authentication NOT set in SecurityContextHolder → filterChain.doFilter() still called (filter doesn't throw) → AnonymousAuthenticationFilter: sets Anonymous authentication → FilterSecurityInterceptor: route requires authenticated() → no auth found → Throws: AccessDeniedException or AuthenticationException ExceptionTranslationFilter catches: → AuthenticationException → AuthenticationEntryPoint → HTTP 401 → AccessDeniedException → AccessDeniedHandler → HTTP 403 CRITICAL: @ControllerAdvice / GlobalExceptionHandler CANNOT intercept this! The request never reached DispatcherServlet! Gateway-level JWT failure (JwtAuthenticationFilter in Gateway): → onError(exchange, "Invalid or expired token", HttpStatus.UNAUTHORIZED) → exchange.getResponse().setStatusCode(401) → exchange.getResponse().setComplete() ← terminates reactive chain PATH B: EXCEPTION IN CONTROLLER/SERVICE LAYER (after DispatcherServlet) ═════════════════════════════════════════════════════════════════════════ Example: User not found, bad credentials, validation failure DispatcherServlet calls Controller → Controller calls Service → Service throws: RuntimeException("User not found") BadCredentialsException ResourceNotFoundException GlobalExceptionHandler (@RestControllerAdvice catches all): File: com.skillsync.auth.exception.GlobalExceptionHandler @ExceptionHandler(ResourceNotFoundException.class) → 404 @ExceptionHandler(MethodArgumentNotValidException.class) → 400 @ExceptionHandler(RuntimeException.class) → 400 @ExceptionHandler(Exception.class) → 500 All return standardized JSON: { "timestamp": "2024-01-01T12:00:00", "status": 400, "error": "BAD_REQUEST", "message": "Email already registered: user@test.com", "path": "/api/auth/register" } FRONTEND HANDLING: Axios response interceptor: → 401 → trigger token refresh → retry → 403 → navigate('/unauthorized') → 5xx → navigate('/500') → 429/503 → exponential backoff retry (max 3 times) → Other errors → Promise.reject → mutation.onError handles + showToast

🔗 Feign Client Calls — Service-to-Service

🛡️ Security Architecture Decisions

DecisionWhat Was ChosenWhyTrade-off
JWT Algorithm HS256 (Symmetric) Auth-service and Gateway are in same trusted infrastructure. Simpler to manage — one shared secret via Config Server. If JWT_SECRET leaks → all tokens forgeable. RS256 would require public/private key pair but allows token verification without the private key.
Token Storage Dual: JSON body + HttpOnly Cookie Cookie for XSS resistance (JS can't steal it). JSON body for cross-domain scenarios where cookie isn't sent (CORS restrictions on cross-subdomain). localStorage is XSS-vulnerable but necessary for cross-domain. HttpOnly cookie is CSRF-vulnerable but XSS-safe. SameSite=None mitigates CSRF here.
Session Policy SessionCreationPolicy.STATELESS No server-side session stored. JWT is self-contained. Scales horizontally without shared session store. Cannot immediately invalidate access tokens. Must wait for expiry (24h). Mitigated by short-lived tokens + refresh token rotation.
Access Token Expiry 24 hours (MIN_ACCESS_EXPIRATION_MS) Longer expiry → fewer refresh calls → better UX. Configured via jwt.access-expiration property. Best practice is 15 min. 24h means a stolen token is valid for up to 24h. No blacklist implemented.
Password Hashing BCrypt with strength=12 BCrypt is slow by design (12 rounds ≈ 250ms on modern hardware). Unique salt per password prevents rainbow table attacks. 12 rounds means login takes ~250ms extra. Mitigated by the fact it only happens at login, not on every request.
CSRF Protection Disabled in auth-service, custom CsrfOriginFilter at Gateway Spring CSRF is designed for session-based apps with forms. JWT is sent in Authorization header — CSRF doesn't apply to header-based auth. Custom origin check at Gateway provides equivalent protection. Refresh token is in a cookie — CSRF could attack /api/auth/refresh. Mitigated by SameSite=None + custom CsrfOriginFilter checking Origin header.
User ID in requests X-User-Id header injected by Gateway ONLY Services trust X-User-Id unconditionally. Gateway strips any client-sent X-User-Id and injects from verified JWT claim. Prevents privilege escalation via header injection. Services have no security layer themselves — they are not safe if accessed directly (bypassing Gateway). Network-level isolation needed in production.

🛡️ CSRF Protection — Actual Implementation

CsrfOriginFilter.java — What It Actually Does
File: api-gateway/src/main/java/com/skillsync/apigateway/filter/CsrfOriginFilter.java @Order(-3) ← runs FIRST before all other filters public Mono filter(ServerWebExchange exchange, GatewayFilterChain chain) { String method = exchange.getRequest().getMethod().name(); if ("OPTIONS".equalsIgnoreCase(method)) { return chain.filter(exchange); ← NEVER block preflight } if (List.of("POST", "PUT", "DELETE", "PATCH").contains(method)) { String origin = exchange.getRequest().getHeaders().getFirst("Origin"); Set allowed = Set.of(allowedOrigins.split(",")); // allowedOrigins = "https://skillsync.udayasri.dev, // http://localhost:5173,http://localhost:3000" if (origin != null && !allowed.contains(origin.trim())) { exchange.getResponse().setStatusCode(HttpStatus.FORBIDDEN); return exchange.getResponse().setComplete(); ← Block immediately } } return chain.filter(exchange); ← GET requests and null Origin pass through } WHAT THIS PROTECTS AGAINST: Malicious site at "evil.com" tries to POST to /api/auth/refresh using victim's cookie: → Browser sends request with Origin: https://evil.com → CsrfOriginFilter: "evil.com" NOT in allowedOrigins → 403 → BLOCKED WHY THIS IS BETTER THAN SPRING'S DEFAULT CSRF: → Spring's default CSRF requires a CSRF token in every form submission → JWT apps don't use form submissions (it's REST API with JSON) → Origin check is simpler and equally effective for the use case here LIMITATION: → null Origin header (direct API calls, Postman, curl) PASSES THROUGH → Acceptable: attacker needs to forge a browser request (can't control Origin in real browsers) → A null origin from an actual browser would mean same-origin → safe AUTH-SERVICE CSRF CONFIG: .csrf(AbstractHttpConfigurer::disable) → CSRF disabled inside auth-service too → Same reasoning: JWT-based, stateless → Gateway's CsrfOriginFilter handles origin validation before reaching auth-service

❓ Advanced Interview Q&A — Part 2

RabbitMQ & Events

QuestionAnswer (from actual code)
Why use RabbitMQ instead of direct HTTP calls for notifications? Decoupling. If notification-service is down when a session is booked, session-service still succeeds. The event stays in the queue (durable=true) and notification-service processes it when it comes back up. With direct HTTP, session creation would fail if notification-service is unavailable.
What is the Outbox Pattern and why is it used in PaymentService? Without Outbox: paymentRepository.save() succeeds, then rabbitTemplate.send() fails → event lost, session never confirmed. With Outbox: both payment record and outbox event are written in the SAME database transaction. OutboxPublisher polls and publishes — if it crashes mid-publish, unprocessed events stay in outbox with published=false → retried. Guarantees at-least-once delivery.
What happens if a RabbitMQ consumer throws an exception? From PaymentEventConsumer.java: throw e; is explicit — re-throwing causes RabbitMQ to NACK the message. If a DLQ is configured (email.retry.queue has DLX), the message goes to the dead letter queue after exhausting retries. The DlqReplayController provides an admin endpoint to replay failed events.
Why is Jackson2JsonMessageConverter configured as a Bean? RabbitMQ's default serialization is Java object serialization (binary, class-dependent, fragile across versions). Jackson2JsonMessageConverter uses JSON — language-agnostic, readable, and forward-compatible. Both sender and receiver must have the same converter configured.

CQRS & Architecture

QuestionAnswer (from actual code)
Why split command and query services if they share the same database? This is "logical CQRS" — same DB, separate service classes. Benefits: command services are @Transactional with cache eviction; query services are read-optimized with Redis caching. If scale demands, can later add read replicas for query services without changing the CQRS interface. Also simpler to test and reason about.
What does cacheService.evictByPattern("v1:session:learner:101:*") do? Redis SCAN operation: iterates all keys matching the pattern using a cursor (not KEYS — which blocks Redis). Deletes all paginated session list caches for learner 101. This is necessary because a session state change (accepted, completed) invalidates ALL cached pages of that user's sessions, not just one key.
What is the vKey() method in CacheService? public static String vKey(String rawKey) { return KEY_VERSION + ":" + rawKey; } → prefixes all cache keys with "v1:". When the cache schema changes (e.g., ProfileResponse gains a new field), increment KEY_VERSION to "v2:" → all old "v1:" caches become stale and are effectively ignored without needing to flush Redis manually.

Session & Mentor Logic

QuestionAnswer (from actual code)
Why does createSession() call Feign to validate the mentor? Two reasons: (1) validate the mentorId is actually a ROLE_MENTOR user — a learner could try to book with any userId. (2) Handle the ambiguity between mentorProfileId (from user-service) vs mentorUserId (from auth-service). resolveMentorUserId() first tries auth-service directly, then falls back to user-service's mentor profile to map the profile ID to a user ID.
What is session.meetingLink set to after payment confirmation? From confirmSessionPayment(): "https://meet.jit.si/SkillSync-" + session.getId() + "-" + System.currentTimeMillis(). Uses Jitsi Meet (free, open-source video conferencing) with a unique room name per session. No account needed — just join the URL.
What does defaultRatingApplied=true mean on session completion? From completeSession(): marks that the session completed and a default rating should be applied if the learner doesn't submit a review. This flag allows a background job to auto-apply a neutral rating after a timeout, preventing mentors from being rated 0 by default for completed sessions without reviews.

Interview-Ready Summary

"SkillSync is a production-grade distributed SaaS platform built on a Cloud-Native Microservices architecture. It uses Spring Cloud Gateway (Netty) as a reactive API Gateway that validates HS256 JWT tokens, performs rate limiting via a sliding-window ConcurrentHashMap, enforces CSRF origin checking, and injects verified user identity as X-User-* headers into downstream requests. The Auth-Service implements a full Spring Security Filter Chain with a custom OncePerRequestFilter for JWT validation, BCrypt-12 password hashing, two-factor OTP verification (SecureRandom, 5-min expiry, 5-attempt limit), Google OAuth linking, and stateful refresh token rotation with FIFO eviction (max 5 per user). All business services follow CQRS — CommandService for transactional writes with cache eviction and RabbitMQ event publishing, QueryService for Redis-cached reads with cache-aside and stampede protection. The Payment flow uses a Saga pattern with the Outbox Pattern for at-least-once delivery guarantee, Razorpay HMAC verification, and automatic session confirmation or compensation via RabbitMQ consumers. Real-time notifications are pushed via STOMP WebSocket through the notification-service. Service discovery uses Netflix Eureka; centralized configuration uses Spring Cloud Config Server; inter-service communication uses declarative Feign clients on internal /api/*/internal/** paths blocked from the Gateway. The React frontend uses Redux Toolkit for state, Axios interceptors for silent JWT refresh with a concurrent-request queue mechanism, and dual token storage (Redux + localStorage for cross-domain, HttpOnly cookies for XSS resistance)."
