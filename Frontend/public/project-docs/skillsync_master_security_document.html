1. The Big Picture — What Is SkillSync?

SkillSync is a distributed SaaS platform built as a collection of microservices. Think of it like a large shopping mall where every shop is independent, but they all share the same entrance gate (API Gateway), the same security company (Auth Service), and the same customer database.

3. The Full Story — "User Clicks Login"

Let's trace every single thing that happens when a user types their email and password and hits Login. Think of this like a forensic detective following a package from sender to recipient.

  • 1
    User clicks Login in the browser

    Login.tsx captures the email and password from the form. It calls handleSubmit().

  • 2
    Redux dispatches the login action

    authSlice.ts has a special async function called loginUser (an Async Thunk). It starts the API call and tells Redux: "Login is in progress."

  • 3
    Axios sends the POST request

    The configured apiClient sends POST /api/auth/login with JSON body {"email":"...","password":"..."}. The request interceptor checks for a token — but since this is a login request, there's nothing to attach yet.

  • 4
    DNS Resolution & Network

    api.skillsync.udayasri.dev resolves to the IP of the Load Balancer or API Gateway. The TLS (HTTPS) handshake happens — the browser and server negotiate a symmetric session key via asymmetric encryption. All data from this point is encrypted.

  • 5
    API Gateway receives the request (Netty)

    Unlike normal Spring Boot (Tomcat), the Gateway uses Netty's event loop model — a single thread handles many connections using non-blocking I/O. The Gateway's JwtAuthenticationFilter runs. Because /api/auth/login is a public path, it is allowed through without token validation.

  • 6
    Request reaches Auth-Service (Tomcat)

    Tomcat's Connector (Coyote) accepts the TCP connection. A worker thread is pulled from the thread pool (e.g., http-nio-8081-exec-3). The CoyoteAdapter converts raw bytes into a proper Java HttpServletRequest object.

  • 7
    Spring Security Filter Chain runs

    Before the request reaches any controller, it must pass through the Security Filter Chain. Filters run in order (covered in Section 5). The JwtAuthenticationFilter sees no Authorization header for the login path and allows it through.

  • 8
    DispatcherServlet routes the request

    The "Front Controller." It asks HandlerMapping: "Who handles POST /api/auth/login?" Answer: AuthController.login(LoginRequest). Spring uses Java reflection to call this method.

  • 9
    AuthController receives the request

    The JSON body is deserialized into a LoginRequest DTO (Data Transfer Object). The controller calls authService.login(loginRequest).

  • 10
    AuthService delegates to AuthenticationManager

    It wraps the credentials in a UsernamePasswordAuthenticationToken and calls authenticationManager.authenticate(token). This is the key delegation step — Spring's security system now takes over.

  • 11
    ProviderManager → DaoAuthenticationProvider

    The ProviderManager iterates through its list of providers. It finds DaoAuthenticationProvider, which knows how to handle username/password credentials.

  • 12
    UserDetailsServiceImpl → Database query

    DaoAuthenticationProvider calls your UserDetailsServiceImpl.loadUserByUsername(email). This queries AuthUserRepository.findByEmail(email) in PostgreSQL. It returns a UserDetails object containing the hashed password and roles.

  • 13
    BCrypt password comparison

    BCryptPasswordEncoder.matches(rawPassword, hashedPassword) is called. BCrypt re-hashes the input and compares it to the stored hash. Why BCrypt? It's slow by design (12 rounds) to resist brute-force attacks, and it includes a random salt so identical passwords have different hashes.

  • 14
    JWT tokens generated

    On success, JwtTokenProvider.generateAccessToken(userId, email, role) creates a signed JWT using HS256. A refresh token is also created and saved to the database. Spring Security automatically nulls out the password from the Authentication object at this point — a safety feature.

  • 15
    Response returned to browser

    The controller returns ResponseEntity.ok(authResponse) with the JWT in the JSON body. The refresh token is set as an HttpOnly, Secure, SameSite=None cookie via Set-Cookie header. HttpOnly means JavaScript cannot read it — preventing XSS theft.

  • 16
    Browser stores tokens, Redux updates

    The access token is stored in localStorage (ss_access_token) and Redux state. The Redux authSlice sets isAuthenticated = true. The UI re-renders showing the dashboard.

  • 17
    Thread cleanup

    After the response is sent, Spring's SecurityContextPersistenceFilter clears the ThreadLocal. The Tomcat worker thread returns to the pool — clean and ready for the next request.

4. Tomcat & Servlet Internals

🏭
Tomcat = A Post Office

Tomcat receives raw letters (TCP packets), opens the envelope (parses HTTP bytes), and hands the sorted letter to the correct department (your Spring controllers). It also manages a pool of workers (threads) to handle many letters simultaneously.

Tomcat's Internal Flow

TCP Bytes Arrive at Port 8080/443 │ ▼ NioEndpoint (accepts TCP connection) → Picks a worker thread from the thread pool → e.g. "http-nio-8081-exec-3" │ ▼ Http11Processor → Reads the raw HTTP bytes off the socket │ ▼ CoyoteAdapter.service() → Converts internal Coyote request → HttpServletRequest (Servlet API) → Converts internal Coyote response → HttpServletResponse │ ▼ Servlet Container Routing: Engine → Host (api.skillsync.udayasri.dev) → Context (auth-service app) │ ▼ Filter Chain (Servlet Filters run BEFORE any controller) │ ▼ DispatcherServlet (Spring MVC Front Controller) │ ▼ Your @RestController method executes │ ▼ Response written back to HttpServletResponse → TCP socket

Filter vs. Interceptor — Why Does It Matter?

Key Distinction

Filters are part of the Servlet Container (Tomcat). They run before Spring MVC even knows the request exists. Security filters live here because we want to block unauthorized requests at the gate before wasting memory on controller logic.

Interceptors are part of Spring MVC. They run after the DispatcherServlet has already routed the request to a controller. They can add logging, audit trails, etc.

Analogy: Filters = Security guard at the building entrance. Interceptors = The receptionist inside the office who checks your appointment before letting you into a specific room.

5. DelegatingFilterProxy — The Bridge

🌉
The Bridge Between Two Worlds

Tomcat lives in one world (Servlet Container). Spring Security lives in another (Spring Application Context). Tomcat doesn't know about Spring Beans. DelegatingFilterProxy is the bridge — it is a standard Servlet Filter that Tomcat recognizes, but internally it delegates all work to the Spring Security filter chain.

  • 1
    Tomcat calls DelegatingFilterProxy

    This is a plain old Servlet Filter that Tomcat registers normally.

  • 2
    Proxy looks up springSecurityFilterChain Bean

    It asks the Spring Application Context: "Give me the Bean named springSecurityFilterChain."

  • 3
    Control passes to FilterChainProxy

    This is the real Spring Security entry point. It holds your list of SecurityFilterChain beans and runs them in order.

6. Spring Security Filter Chain — Step by Step

Every request passes through these filters in order. Each filter does its job and calls chain.doFilter() to pass the request to the next filter. If a filter decides to block the request, it does NOT call chain.doFilter() — the request stops dead and a response is sent immediately.

⚠️
Critical Rule If any filter does NOT call chain.doFilter(), the request is halted. The user sees an error response (401 or 403). The Controller is NEVER reached.

The Authorization Vote System

AccessDecisionManager — The Voting System

When the FilterSecurityInterceptor checks permissions, it uses an AccessDecisionManager. The default is AffirmativeBased — like a democracy where if even ONE voter says YES, access is granted. Each voter checks one thing (e.g., "Does the user have ROLE_MENTOR?").

8. JWT — Your Digital Concert Wristband

🎫
JWT = Concert Wristband

You show your ID at the ticket booth (Login). They give you a wristband (JWT). For the rest of the night, security at every stage doesn't ask for your ID — they just check your wristband. Even if there are 10 different stages (microservices), the wristband works everywhere. If the wristband expires or is fake, you're denied.

JWT Validation in JwtAuthenticationFilter

  • 1
    Extract token

    Read Authorization header. Strip "Bearer " prefix: bearerToken.substring(7). Also checks the accessToken cookie as a fallback.

  • 2
    Validate signature

    The jjwt library re-computes the HMAC-SHA256 signature using the stored JWT_SECRET and compares it to the token's signature. If they match, the token is genuine and unmodified.

  • 3
    Check expiry

    The exp claim is compared to the current time. If expired, an ExpiredJwtException is thrown → 401 response.

  • 4
    Load user details & populate SecurityContext

    If valid, it extracts the email from claims, calls UserDetailsServiceImpl.loadUserByUsername(email), creates a UsernamePasswordAuthenticationToken, and sets it in SecurityContextHolder. The request is now "authenticated."

HS256 vs RS256 — Why Symmetric?

Design Decision: HS256

SkillSync uses HS256 (Symmetric) — both the Auth Service and the API Gateway share the same secret key. This works because both services are inside our private infrastructure (same trust boundary). It's faster and simpler.

RS256 (Asymmetric) would be needed if external third-party services needed to verify our tokens — they could use the public key without ever knowing the private key. For SkillSync's internal cluster, this is unnecessary overhead.

9. ThreadLocal & SecurityContext — The Invisible Backpack

🎒
ThreadLocal = An Invisible Backpack

Imagine each worker (thread) at the office has an invisible backpack. When they pick up a task (request), the security guard puts the user's identity card in their backpack. As the worker walks through different rooms (Controller, Service, Repository), they always have the identity card on them. When they're done and return to the waiting area (thread pool), the guard empties the backpack — so the next task doesn't accidentally find someone else's identity card in there.

Technical Detail

SecurityContextHolder Internals

SecurityContextHolder is a static utility class. Internally, it uses a ThreadLocal<SecurityContext>. ThreadLocal is a Java feature that creates a separate "slot" for each thread — Thread 101 and Thread 102 both have a ThreadLocal variable, but they each see different values. It's like each thread has its own private copy of a variable.

Lifecycle

Request arrives → Thread 101 picked from pool ↓ JwtAuthenticationFilter validates JWT ↓ SecurityContextHolder.getContext().setAuthentication(user) [Thread 101's ThreadLocal now holds User A's identity] ↓ Controller reads: SecurityContextHolder.getContext().getAuthentication() Service reads: SecurityContextHolder.getContext().getAuthentication() [All code in Thread 101 can access User A's identity — no need to pass it as parameters] ↓ Response sent ↓ SecurityContextPersistenceFilter CLEARS the ThreadLocal [Thread 101 is now clean — no identity left in backpack] ↓ Thread 101 returns to pool — ready for the next user
🚨
Thread Poisoning — Critical Bug If Cleanup Fails If the ThreadLocal is NOT cleared and Tomcat reuses Thread 101 for a different user's request, the new user would be authenticated as the OLD user. This is a catastrophic security bug. Spring prevents it automatically via SecurityContextPersistenceFilter.
⚠️
Manual Threads Don't Inherit SecurityContext If you create a new Thread() inside a service, that new thread does NOT have the SecurityContext. Calling SecurityContextHolder.getContext().getAuthentication() there returns null → NullPointerException. Solution: Use SecurityContextHolder.MODE_INHERITABLETHREADLOCAL or pass the Authentication explicitly.

10. API Gateway — The Security Checkpoint

Gateway vs. Service — Double Gating

SkillSync uses a "Double Gate" pattern. The Gateway validates JWT signature (cryptographic check — fast, no DB). The Auth Service does the full credential check (DB query — only on login). This means for every API call after login, the DB is NOT queried for authentication — only the gateway's signature check runs. This is what makes microservices scalable.

  • 1
    Netty picks up the TCP packet

    The Gateway uses Netty's event loop — a single-threaded, non-blocking model that handles thousands of connections efficiently. Much lighter than Tomcat's thread-per-request model.

  • 2
    JwtAuthenticationFilter (Reactive) validates token

    Extracts JWT, validates HMAC-SHA256 signature using the shared secret. Extracts sub (userId), email, and role claims.

  • 3
    Request Mutation (Critical Security Step)

    The Gateway does TWO things: (1) STRIPS any incoming X-User-Id header — preventing a malicious client from forging their identity. (2) INJECTS the verified X-User-Id header from the JWT claims. Downstream services can trust this header completely.

  • 4
    Route to downstream service via Eureka

    The Gateway uses LoadBalancerClient to look up the target service in Eureka. It forwards the mutated request with the injected headers.

  • 5
    Internal path blocking

    The Gateway explicitly blocks any request to /api/auth/internal/** from external users. These paths are only for service-to-service internal calls.

12. Frontend — The Self-Healing UI

Axios Interceptors

Request Interceptor — Auto-Attach JWT

Before every API call, the Axios request interceptor reads the accessToken from Redux state and attaches it as Authorization: Bearer <token>. The developer never manually adds this header — it happens automatically for every request.

Response Interceptor — Silent Token Refresh (Self-Healing)

If any API call returns 401 Unauthorized (token expired), the response interceptor automatically:

  1. Calls /api/auth/refresh using the HttpOnly cookie (refresh token)
  2. Gets a new access token
  3. Updates localStorage and Redux state
  4. Retries the original failed request with the new token

The user never sees a flicker or a logout. This is the "Self-Healing" behavior.

💡
The Stateless Paradox SkillSync uses stateless JWT access tokens (no server storage needed) but stateful refresh tokens (stored in PostgreSQL). This combination gives the best of both worlds: access tokens are fast (no DB hit), and refresh tokens can be revoked (delete from DB to force logout).

18. Refresh Token Rotation

🔄
Refresh Token = Library Card

Your access token (concert wristband) expires after 24 hours. Your refresh token is like a library card — you use it to get a new wristband. But here's the trick: every time you use the library card, you get a NEW library card and the old one is destroyed. If someone stole your library card, they can only use it once before it's invalidated.

Access Token expires (24h) ↓ Axios Response Interceptor catches 401 ↓ POST /api/auth/refresh (refresh token sent via HttpOnly cookie) ↓ AuthService.refreshToken(): 1. Find refresh token in PostgreSQL by token value 2. Check: is it expired? (7-day window) 3. DELETE old refresh token from DB ← Rotation step 4. Generate new Access Token 5. Generate new Refresh Token and save to DB 6. Return new Access Token in response + new Refresh Token in cookie ↓ Axios retries original failed request with new Access Token
⚠️
Current Security Gap SkillSync does NOT implement "sliding window detection." If a refresh token is stolen, the attacker can keep refreshing for up to 7 days. Single-use rotation (deleting on use) limits the window, but does not eliminate it. A future improvement would detect concurrent refresh attempts (detecting that the token was already used once) and force a full logout.

19. CSRF & CORS Explained Simply

CSRF — Why It's Disabled in SkillSync

🎭
CSRF = Tricking someone into performing an action

Imagine a malicious website has a hidden button that, when clicked, secretly transfers money from your bank. If your bank uses sessions (stored in cookies), the malicious site can forge requests on your behalf. CSRF protection prevents this.

Why disabled in SkillSync? JWT is sent in the Authorization header — not in a cookie. Malicious sites cannot access HTTP headers of cross-origin requests (browser blocks this). So CSRF is not a threat when using JWT in headers.

CORS — Cross-Origin Resource Sharing

CORS Configuration

The browser's same-origin policy blocks JavaScript from reading responses from different domains. CORS is how the server tells the browser: "It's okay for skillsync.udayasri.dev to read my responses."

SkillSync configures CORS at the Gateway level — allowing specific origins while blocking others. The CorsFilter handles preflight OPTIONS requests that browsers send before the actual request.

20. Interview Q&A — Wh-Questions

Tomcat & Servlet

What is DelegatingFilterProxy and why does it exist?
It is a standard Servlet Filter that acts as a bridge between Tomcat and Spring Security. Tomcat cannot directly access Spring Beans. DelegatingFilterProxy is a Tomcat-compatible filter that looks up the springSecurityFilterChain bean inside the Spring ApplicationContext and delegates all security work to it. Without it, Spring Security couldn't participate in Tomcat's filter chain.
What happens if chain.doFilter() is not called in a filter?
The request is immediately halted. The response is sent back to the client right there in that filter. The Controller (DispatcherServlet) is never reached. This is how security filters block unauthorized requests — by NOT calling chain.doFilter() when a JWT is invalid or missing.
What is the difference between a Filter and an Interceptor?
Filters are part of the Servlet Container (Tomcat) and run before Spring MVC processes the request. Interceptors are part of Spring MVC and run after the DispatcherServlet routes the request to a controller. Security must use Filters because we want to block access before spending resources on controller logic.
Why does Tomcat use a thread pool?
Creating a new OS thread for every request is expensive (time and memory). A thread pool keeps threads alive and reuses them. When a request arrives, a pre-existing thread is assigned. When done, the thread returns to the pool. This is why ThreadLocal MUST be cleared — otherwise the same thread serving a new user would still carry the previous user's security context.

Spring Security

Who clears the SecurityContext after each request?
The SecurityContextPersistenceFilter (or its modern equivalent) ensures the ThreadLocal is cleared after the response is sent. This prevents thread poisoning — where Thread A's security context bleeds into Thread B's next request.
What is the SecurityContextHolder and why use it?
It is a static utility class that stores the current user's Authentication object in a ThreadLocal variable. Without it, every method in the call stack (Controller → Service → Repository) would need to accept the User object as a parameter — "parameter pumping." SecurityContextHolder makes the current user globally accessible within the current thread without polluting every method signature.
What happens when UsernamePasswordAuthenticationFilter is removed?
Nothing — SkillSync intentionally doesn't use it. This filter handles Spring's default form-based login. SkillSync replaces it with a custom JSON login endpoint in AuthController that gives more control over the login response (returning tokens in both JSON body and HttpOnly cookie).
What is DaoAuthenticationProvider and what does it do?
It is Spring Security's default credential checker. It calls UserDetailsService to load the user from the database, then uses the configured PasswordEncoder (BCrypt) to verify the raw password against the stored hash. On success, it returns a fully populated Authentication object. Spring automatically nulls the password from this object for safety.

JWT

What if the JWT secret is leaked?
Critical impact. Anyone with the JWT secret can forge tokens for any user, including Admin. An attacker could create a token with role=ROLE_ADMIN and bypass all security. Action: immediately rotate the JWT_SECRET environment variable across all services (Gateway, Auth-Service). All existing tokens become invalid because they were signed with the old secret. Users are forced to log in again.
Why use stateless JWT but stateful refresh tokens?
This is the "Stateless Paradox." Access tokens are stateless (fast — no DB check) but cannot be revoked before expiry. Refresh tokens are stateful (stored in DB) so they CAN be revoked (delete the row). This gives us speed for normal operations and revocability for security events like logout or account suspension.
Why is HS256 used instead of RS256?
Because the Auth Service and API Gateway are in the same private infrastructure (same trust boundary). HS256 is symmetric — one shared secret — which is faster and simpler. RS256 (asymmetric) would be needed if external partners needed to verify tokens without knowing our private key. For an internal microservice cluster, HS256 is the correct and more efficient choice.
Can a JWT be revoked before it expires?
Not in the current SkillSync implementation. Once an access token is issued, it is valid until it expires (24 hours) — this is the fundamental weakness of stateless JWT. The planned solution is a Redis blacklist: on logout, store the token's JTI (unique ID) in Redis with a TTL matching the remaining token lifetime. The JwtAuthenticationFilter checks if the JTI is blacklisted on every request.

Architecture

Why does the User-Service trust the X-User-Id header?
Because only the API Gateway can inject that header. The Gateway first strips any X-User-Id the client sends (preventing forgery), then injects the verified X-User-Id extracted from the JWT. Since the User-Service is not directly accessible from the internet (only via Gateway), it can safely trust that any X-User-Id header it receives was verified by the Gateway. This makes User-Service "security-agnostic" — it focuses purely on business logic.
What happens if auth-service goes down?
Eureka detects the missing heartbeat within seconds and marks auth-service as unavailable. The Gateway stops routing traffic to it. New logins would fail. However, existing sessions (valid JWTs) would continue to work for protected routes, since the Gateway validates JWT without contacting auth-service. The system degrades gracefully — only login/refresh would fail.
What is Eureka and why is it used?
Eureka is a Service Discovery registry. Instead of hardcoding IP addresses for each microservice, services register themselves with Eureka on startup with a heartbeat. The Gateway uses Eureka to find the current healthy instances of any service by name. This enables load balancing and automatic failover — if one instance of user-service goes down, the Gateway routes to another healthy instance automatically.

Master Summary — One Paragraph

"SkillSync is a full-stack distributed microservices platform. It uses stateless JWT-based authentication (HS256) enforced at the API Gateway (Netty), with a centralized Auth-Service (Tomcat/Spring Security) managing JWT lifecycle and Refresh Token Rotation (database-backed). The React frontend is a 'Self-Healing UI' using Axios interceptors for silent token refresh and Redux Toolkit for real-time state management. Tomcat manages threads via a thread pool + ThreadLocal (SecurityContext) pattern — each request carries the authenticated user's identity in an invisible backpack that is cleared after the response. The system is horizontally scalable because downstream services (User-Service, Skill-Service) never touch the database for authentication — they simply trust the X-User-Id header injected by the Gateway."
