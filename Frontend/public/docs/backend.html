System Components explained

The Microservice Codex

Every domain within SkillSync operates with absolute autonomy. Each microservice manages its own separate PostgreSQL database layer and scales completely independently to address its unique traffic loads. Listed below is bit-by-bit coverage of their operations.

🔑 Auth Service

The cryptographic gatekeeper. Responsible for securely hashing passwords, validating external Google OAuth payloads, and issuing the signed JWTs that physically unlock the rest of the private network.

APIs & Services Exposed

  • POST /auth/login - Native credential verification
  • POST /auth/google - OAuth2 Payload unwrapping natively
  • POST /auth/refresh - Minting new access tokens from Redis
  • POST /auth/reset-password - Secure email reset dispatch

Internal State & Dependencies

  • Schema: Completely isolated auth_db PostgreSQL ownership
  • Cache: Stores active refresh tokens centrally in the Redis Edge Cluster
  • Sync REST: Pings User Service on first Google Login to explicitly create missing profiles
Boundary & Operating Radius Auth Service strictly manages ONLY cryptographic identity and authentication states. It operates within a rigid radius. It is explicitly forbidden from maintaining user profiles, handling payments, or storing skill tags. Once it verifies identity and signs the JWT, its job is entirely complete. It does not control authorization (which Gateway handles).

👥 User Service

The core central authority. It houses the massive operational logic detailing mentor applications, learner profiles, study group formations, and the complex authorization linkages.

APIs & Services Exposed

  • GET /users/profile - Retrieves public user topologies
  • POST /users/mentors - Mentor application initiation
  • PUT /users/admin/approve - Admin-level mutation of roles
  • POST /groups - Study group logic logic creation

Internal State & Dependencies

  • Schema: Maintains heavy read-write bounds over user_db
  • CQRS Cache: Mirrors its group schemas and active profiles directly into Redis for massive-scale zero-latency reads.
Boundary & Operating Radius The User Service handles anything dealing with physical profiles, role assignment data, or social grouping. It acts as the "source of truth". However, its radius ends at social entities. It cannot process payments, and it cannot directly log someone in. It solely houses data relating to WHO the subject is.

📚 Skill Service

The domain catalog mapping thousands of technology topics into categorized trees. It provides lightning-fast paginated search endpoints so learners can discover Mentors based on exact, standardized topic strings.

APIs & Services Exposed

  • GET /skills - Master catalog acquisition
  • GET /skills/search - Deep indexed keyword searches
  • GET /skills/categories - Taxonomic hierarchical returns

Internal State & Dependencies

  • Schema: Maintains extreme index depths in skill_db
  • Isolation: Requires essentially zero cross-talk with other services. Fully decoupled utility.
Boundary & Operating Radius The Skill Service operates with the smallest, most defined radius on the platform. It literally only cares about the existence of tags (e.g., "React", "Docker"). It does NOT know which users have which skills (that is the User Service's radius). It simply acts as the master dictionary for the system.

📅 Session Service

The most transactionally sensitive application on the platform. Exists entirely to prevent double-booking, lock physical time-slots, enforce Mentor availability caps, and aggregate reviews.

APIs & Services Exposed

  • POST /sessions/book - Initializes mentorship saga
  • PUT /sessions/schedule - Mutates availability matrices
  • POST /sessions/reviews - Locks in post-session ratings

Internal State & Dependencies

  • Schema: Operates session_db with strict locking
  • Async Sagas: Publishes session.pending to RabbitMQ allowing Payment Svc to activate asynchronously.
  • Sync Ping: Validates Mentor explicitly with User Svc before allowing a pending booking.
Boundary & Operating Radius The Session Service radius controls "Time". If an entity wishes to occupy time, it must go through Session logic. It locks availability globally but it delegates financial operations. It operates in PENDING modes, waiting for external systems (RabbitMQ) to finalize its states.

💳 Payment Service

The financial ledger. Operates completely in the background without UI intervention. It takes abstract events, communicates exclusively with Razorpay's API, and listens endlessly for Webhook callbacks.

APIs & Services Exposed

  • POST /payments/init - Internal initialization protocol
  • POST /payments/webhook - Razorpay's external asynchronous callback ping targeting our backend
  • POST /payments/refund - Saga compensation routine

Internal State & Dependencies

  • Schema: payment_db functions as an append-only write ledger mapping transaction IDs
  • Saga Broadcaster: Listens to session.pending and natively broadcasts payment.success and payment.failed backwards out via RabbitMQ to finish the transaction.
Boundary & Operating Radius The Payment Radius strictly bounds money. The Payment service does not know what a "Session" or "Mentorship" is. It only knows what an OrderID and an Amount is. Once Razorpay processes it, Payment Svc drops a generic success token into the bus, completely blind to what the downstream services actually do with it.

🔔 Notification Service

The communications hub. A service that functions as a pure real-time data terminal. It consumes messaging events and fires them out of the platform via Emails and TCP pipe pushes.

APIs & Services Exposed

  • WS /ws - The primary STOMP WebSocket entrance
  • GET /notifications/unread - Fetches offline history
  • PUT /notifications/read - Mutates badge state

Internal State & Dependencies

  • Schema: notify_db manages fast ephemeral "Read/Unread" booleans
  • Protocols: Maintains native JavaMailSender pools for SMTP dispatches.
  • Broker: Operates an in-memory SimpleBroker pushing WebSockets.
Boundary & Operating Radius The Notification Service radius strictly manages delivery. It has absolutely no decision-making power. It does not create content, it solely relays it. If Session says "booking complete", Notification sends the email. If User Svc says "group chat sent", Notification immediately broadcasts it to the active sockets.
Role Topography

Jurisdiction & Radius of Work

Every request passing through the API Gateway yields its JWT token. The Gateway extracts the embedded Role claims, injects `X-User-Role` headers, and enforces rigid architectural boundaries globally. No service trusts the client; they only trust the Gateway.

✦

The Learner

Radius: Consumer / Explorer

Learners sit at the base of the access tree. They possess sweeping GET permissions to traverse the application, but their POST radii are strictly isolated to their own IDs.

  • Can query `Skill Svc` to search thousands of topics via Redis mappings.
  • Can execute `POST /sessions/book` triggering the Saga to pay a mentor.
  • Can establish WebSockets to chat inside study groups they belong to.
  • Cannot access mentor payout algorithms or modify system categories.
✧

The Mentor

Radius: Creator / Earner

Mentors operate within an elevated business logic layer. Their token claims grant direct write access to scheduling grids and group ownership logic.

  • Can hit `PUT /sessions/schedule` to mutate global availability matrices.
  • Can execute `POST /groups/manage` yielding admin rights over specific chats.
  • Can read sensitive feedback loops bypassing standard learner views.
  • Cannot unilaterally approve other potential mentors.
★

The Administrator

Radius: Global Supervisor

Admins hold overriding absolute logic access. The Gateway specifically permits their tokens past the highly restricted `/admin/**` endpoint layers across all services.

  • Can mutate the catalog hitting `/admin/skills` to permanently add/delete topics.
  • Can approve or reject queued Mentor Applications bypassing automated checks.
  • Can access global system analytics sweeping across the entire platform.
  • Can force-cancel active sagas if dispute resolution is required.

Authentication Pipeline

The JWT Issuance Pipeline

How the physical JSON Web Token is minted and dispersed across the Redis Cluster.

Client Post Gateway Filter Auth Service Redis Store

1. Intercept

The Spring Cloud Gateway catches `/api/**`. If there is no token, it hard rejects `401 Unauthorized`.

2. Vault Operations

Auth Service verifies Google payload, hits `auth_db`, mints secure JWT keys, and lodges identity in Redis.

3. Identity Headers

Gateway unwraps the token and strictly injects `X-User-Role` downstream locking the radius per user type permanently.

Broker Mechanics

Asynchronous CQRS & WebSockets

The Social Synchronization

How group creations and social chats flow without bogging down the database.

  • ◆ CQRS Read Segregation When a Mentor updates a group, `user_db` writes the truth, but simultaneously fires a sync to `Redis`. Learners querying `/groups` hit Redis for massive scale zero-latency reads.
  • ◆ STOMP In-Memory Routing Live Chats do not touch SQL. Messages push to `Notification Svc` which acts as a pure TCP router, broadcasting frames directly to active connected sockets.

Distributed Transactions

The Mentorship Saga Engine

When money is involved across distributed networks, we deploy the Saga Pattern—a heavily regulated, state-machine driven event chain linked solely via RabbitMQ.

The Golden Rule

If the Razorpay webhook fails, the Payment Service emits `payment.failed`. The Session Service consumes this and automatically fires a compensation operation—cancelling the mentorship, unbooking the slot, and ensuring absolute data consistency.

1

Initialization (Session Svc)

Validates mentor limits with User Svc. Creates `session_record` as PENDING(1). Drops `session.requested` block into RabbitMQ.

2

Financial Ledgering (Payment Svc)

Picks up request. Reaches out to Razorpay externally. Locks ledger states. Awaits asynchronous Webhook resolution.

3

Settlement & Delivery (Notify Svc)

Upon `payment.success` event via RabbitMQ, Session explicitly overwrites state to CONFIRMED(3). Notify Svc instantly fires receipt via SMTP JavaMailSender.
